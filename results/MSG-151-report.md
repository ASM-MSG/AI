# MSG-151 — 대규모·동시 처리 부하 측정 (Kafka 도입 판단)

2026-07-22, dev EC2(t3.small, x86_64 2vCPU 1.9GB + 스왑 2GB) 실측.
전체 파이프라인(BE 업로드 → 인코딩 → AI 블러·하이라이트 → READY)을 실제 사용자 경로 그대로 태웠다.
스크립트: `scripts/ec2-load-test.sh` (self-contained, EC2에서 실행).

## 방법

- 샘플: MSG-142의 `crowd.mp4`(pexels 855564, 회랑에 사람 빽빽) 29초 1080p — 티켓의 "사람 많은 30초 영상".
  30초 정각이 아니라 29초인 이유: ffmpeg `-t 30`은 프레임 경계 때문에 30.03초가 나와
  **BE의 30초 상한 검증에 걸린다** (부수 발견 ①, 아래).
- 동시 N건: presigned 발급 → S3 PUT → 업로드 확정까지 N개 병렬. 격자 점령 경합이 노이즈가 되지 않게
  건마다 다른 격자 좌표 사용.
- 관측: DB `processing_status` 15초 폴링 + 메모리·스왑·load 10초 샘플링. AI는 웜 상태
  (모델 로드 완료, N=3 첫 건만 콜드 — 아래 주석).

## 관찰

1. **업로드 경로는 부하와 무관** — 동시 6건에도 확정 API 응답 ~2초. 비동기 설계(MSG-143 ADR) 의도대로
   사용자 대기 시간은 큐 길이와 분리돼 있다.
2. **병목은 AI 단일 워커 큐** — 인코딩은 건당 ~40초로 빠르게 빠지고, AI가 건당 ~2.6분(웜, crowd 기준)으로
   처리량 상한을 결정한다. READY 간격이 그대로 ~2.6분씩 벌어진다. 시간당 처리량 ≈ **20~23건**
   (MSG-143 추정 ~17건보다 소폭 높음 — 720p 인코딩본 입력 덕).
3. **유실 0** — N=3·N=6 전 건 READY 수렴. 큐 적체 중 타임아웃 오탐도 0건:
   MSG-149 D4(큐에 살아있는 QUEUED/PROCESSING 잡은 타임아웃 검사 제외)가 실전에서 검증됨.
4. **t3.small 자원 한계 명확** — 동시 부하 시 load1이 vCPU의 3배, 스왑 사용. 인코딩(ffmpeg)과 AI 추론이
   같은 2vCPU를 나눠 쓰므로 겹치는 구간에서 서로 느려진다.

## 판단: Kafka **불필요** (현 구조로 충분)

- 큐잉 자체는 이미 성립한다: BLURRING 상태가 DB에 남는 폴러 구조(MSG-149 D3)가 재시작 복구·유실 방지를
  보장하고, 이번 실측에서 적체·복구 모두 문제없이 동작했다.
- 문제는 **큐 인프라가 아니라 처리량**이다. Kafka를 넣어도 AI 워커가 1개면 시간당 ~20건은 그대로다.
  처리량을 올리려면 AI 서버 스케일아웃(인스턴스 증설/GPU)이 먼저고, 그때도 다중 컨슈머 분배는
  DB `SKIP LOCKED`(MSG-149 범위 밖 항목)로 충분하다.
- MVP 트래픽(팀 데모·소수 사용자)에서 동시 6건은 이미 극단 케이스다. 마지막 건도 30분 내 READY.
- 재평가 트리거: 시간당 업로드가 지속적으로 20건을 넘거나, AI 워커를 수평 확장해 잡 분배가 필요해질 때.

## 부수 발견

1. **30.0x초 영상은 업로드 거부** — BE가 durationSec≤30 검증과 별개로 실제 길이 30.03초를
   "영상 길이 초과로 인코딩 중단"(FAILED) 처리한다. 촬영·트림 UI가 30초 정각을 만들면 초과분 프레임 때문에
   실패할 수 있다 → FE/BE에 공유 필요 (허용 오차 또는 트림 정책).
2. **모델 콜드 로드 비용** — AI 컨테이너 재시작 후 첫 잡은 모델 로드로 수 분 추가. 배포 직후
   워밍업 요청을 고려할 것.
