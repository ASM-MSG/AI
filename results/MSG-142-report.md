# MSG-142 — AI 처리 시간·리소스 측정 결과

측정일: 2026-07-20 · 도구: [`bench.py`](../bench.py) · 원시 결과: `cpu-*.json`

## 요약

**AWS Graviton3(c7g.large) 실측 기준.** 로컬 M5 노트북 수치는 3.3~4.5배 낙관적이라
인프라 판단에 쓸 수 없어 별도로 실측했다.

- **1080p 30초 → 3~4분, 4K 60fps 30초 → 9.7분.** Lambda 15분 제한의 **65%** (4K 최악)
- **vCPU를 늘려도 거의 안 빨라진다** (2→1 vCPU에서 1~10% 차이). 추론이 병렬화되지 않아
  Lambda 메모리를 올려 vCPU를 늘리는 최적화가 통하지 않는다
- **메모리 4K에서 1,152MB** → Lambda 1024MB 불가, **2048MB 이상** 필요
- **핵심 권고: 4K를 1080p로 다운스케일 후 처리.** 추론은 어차피 `imgsz=640` 기준이라
  정확도 손실이 거의 없는데 9.7분 → 3~4분으로 줄어 마진이 넉넉해진다

## MSG-143(ADR)에 넘기는 입력

| 판단 기준 | Graviton3 실측값 |
|---|---|
| 1건 처리 시간 | 1080p **3.1~3.8분** / 4K 60fps **9.7분** |
| Lambda 15분 제한 대비 | 1080p 21~26% / 4K **65%** — 초과하지 않지만 마진 35% |
| 메모리 피크 | 1080p 886~906MB / 4K **1,152MB** → **2048MB 권장** (1024MB 불가) |
| vCPU 확장성 | **없음** — 2→1 vCPU에서 1~10% 차이뿐 |
| GPU 필요성 | **불필요.** CPU로 제한 안에 들어오고, 다운스케일하면 마진이 더 커진다 |

### Lambda 채택 시 주의

4K가 제한의 65%지만 이 측정에 **포함되지 않은 시간**이 있다:

- S3에서 원본 다운로드 / 처리본 업로드 (4K는 수십 MB)
- 콜드스타트 시 컨테이너 이미지 pull (torch + 모델로 이미지가 크다)
- 모델 로드 (측정에 포함, 1~2초)

이걸 더하면 4K 최악 케이스가 11~13분까지 갈 수 있어 마진이 얇다.

### 권고: 4K를 1080p로 다운스케일 후 처리

`ms/frame`이 해상도와 무관하게 **227~230ms로 일정하다.** ultralytics가 입력을
`imgsz=640`으로 리사이즈하기 때문이다. 즉 4K가 느린 건 추론이 무거워서가 아니라
**프레임 수(60fps)와 인코딩·마스킹 비용** 때문이다.

전처리로 1080p 30fps로 낮추면 프레임 수가 1800 → 900으로 줄어 **9.7분 → 3~4분**이 된다.
추론 정확도는 어차피 640으로 리사이즈되므로 사실상 손실이 없다. 서비스에서 재인코딩은
어차피 하므로 추가 비용도 아니다. **이 전처리를 넣으면 Lambda 마진이 넉넉해진다.**

**최적화 여지**: 4K를 1080p로 다운스케일한 뒤 처리하면 인코딩 비중(24%)이 크게 줄고
프레임당 추론은 어차피 `imgsz=640` 기준이라 정확도 손실도 거의 없다. 어차피 서비스에서
재인코딩을 하므로 추가 비용이 아니다.

## 발견된 문제 (MSG-142 범위 밖, 후속 필요)

**1. 얼굴 미탐지가 심각하다.** 처리본을 육안 확인한 결과, 군중 영상에서 정면 얼굴을
다수 놓쳤고 차량 유리 너머 운전자는 거의 전부 놓쳤다. `conf 0.25`로 이미 낮게 잡은
상태다. MSG-140 완료 조건("얼굴을 자동 감지해 블러")을 현재 설정으로는 만족하지 못한다.
→ conf 하향, 더 큰 모델(YOLOv11s/m), imgsz 상향 중 무엇이 효과적인지 실험 필요.

**2. 단일 씬 영상에서 하이라이트가 0개다.** `plates.mp4`처럼 장면 전환이 없으면
PySceneDetect가 아무 구간도 반환하지 않는다. MSG-141 완료 조건은 "3개 추천"이다.
사용자가 한 자리에서 30초를 찍는 흔한 경우가 여기 해당한다.
→ 장면 전환이 없을 때의 폴백(균등 분할, 움직임량 기준 등)이 필요하다.

## 이 측정의 한계

- **로컬에서 저사양 환경을 흉내내려던 시도는 실패했다 (기록용).**
  `torch.set_num_threads(1)`을 설정해도(`get_num_threads()`는 1을 반환) 실제 연산 속도가
  변하지 않는다 — 2000×2000 matmul 20회가 1스레드 0.15초, 4스레드 0.14초로 동일했다.
  Apple Silicon PyTorch가 Accelerate/AMX 백엔드를 타고 torch 스레드 설정을 무시하기
  때문이다. 그래서 EC2 실측으로 전환했고, 리눅스에서는 `taskset`이 정상 작동한다
  (같은 matmul이 2코어 3.16초 → 1코어 5.95초). 앞으로도 macOS에서 코어 제한 방식으로
  인스턴스 성능을 추정하지 말 것.

- **GPU 측정 미완** — MSG-142 완료 조건의 CPU/GPU 비교 중 GPU 쪽이 남아 있다.
  AWS GPU 인스턴스에서 `--device cuda`로 별도 측정 예정. 다만 CPU만으로 충분하다는
  결론이 이미 나와 GPU 수치는 참고용에 가깝다.
- **한국 번호판 미검증** — 스톡 영상은 유럽 번호판이다. 검출 자체는 작동했으므로
  MSG-144의 "번호판 검출은 국가 무관" 가설을 지지하지만, 한국 번호판 실측은 남아 있다.
- **하이라이트 품질은 평가 불가** — 원본을 `-stream_loop`으로 늘렸기 때문에 장면 전환이
  이음새에서 인위적으로 발생한다. 처리 시간 측정에는 영향이 없다.
- **육안 검증은 프레임 단위 표본** — 미탐지율을 정량 측정하지 않았다. 정확도를 다루는
  후속 작업에서는 라벨링된 데이터셋이 필요하다.
